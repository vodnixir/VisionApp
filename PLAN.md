## 0. Что это за продукт

AR-игра-дуэль. Камера смартфона берёт двух игроков рядом, рисует каждому рамку, измеряет
физическую активность (движение запястий + плеч, нормированное на диагональ тела → **дистанция
до камеры не важна**). У каждого шкала 0–100%. Кто быстрее двигается — быстрее набивает шкалу.
Первый до 100% побеждает. Есть кастинг на ТВ (нативный screen-mirror).

**Стратегическое позиционирование (утверждено):**
> «Это не игра, в которую играешь. Это соревнование, которое ты проводишь.»
> Телефон = цифровой рефери. Целевой сегмент: **семьи + детские аниматоры + школы/лагеря**.

## 1. Текущее состояние кода (факты на 2026-07-08)

Стек: React 19 + TypeScript + Vite 8 + TensorFlow.js pose-detection + Capacitor 8 (Android) + Tailwind 4.

Что реально есть:
- ✅ CV-ядро дуэли (`src/cv/tracking.ts`): скоринг активности по keypoints, EMA-сглаживание,
  дедзона (`MOTION_DEADZONE`), защита от подмены скелета (`TELEPORT_GUARD`), персистентность трека.
- ✅ FSM игры (`src/hooks/useGameState.ts`): SETUP → CALIBRATION → PLAYING → GAME_OVER.
- ✅ Калибровка: обоих надо держать в кадре 3 сек (`LOCK_DURATION_MS`), затем отсчёт 3-2-1.
- ✅ Победа: первый до `targetScore` (100%). Есть обработка «фото-финиша» (оба на 100%).
- ✅ Экран настройки (`SetupScreen.tsx`): имена, сложность, зеркало, звук.
- ✅ HUD, оверлеи калибровки/статуса, экран победы, SFX.
- ✅ Деплой на GitHub Pages (workflow есть).

### ⚠️ Несоответствия кода и брифа (починить):
- «SPRINT / FIGHT / MARATHON» = НЕ таймеры раунда, а **сложность** (`FILL_RATE` easy 9 / normal 6.5 /
  hard 4.5 %/сек). Подпись «~30/60/90 сек» вводит в заблуждение. → Решить: сделать реальные режимы
  с таймером ИЛИ честно переименовать в «Сложность».
- «TV MIRROR» = горизонтальное зеркалирование картинки, НЕ кастинг.

### Чего НЕТ вообще (пробелы):
- Прогрессии, профилей, аккаунтов, рангов.
- Шаринга / записи / реплеев.
- Турнирной сетки.
- Вариативности механик (только «маши быстрее»).
- Форы/хендикапа.
- Локализации (всё English).

## 3. Монетизация (утверждено)

Принцип: **платит взрослый-ведущий, ребёнку — бесплатно и полно. Без энергии/жизней/рекламы в матче.**
- Free (щедрый): полная дуэль, 2–3 режима, сетка на 4, профили. Фора всегда бесплатна.
- Pro/Host (разово ИЛИ подписка): все режимы, сетка 16, ростеры, экспорт, премиум-косметика, без рекламы.
- Косметика IAP: чисто декоративно.
- B2B/SaaS: посадочные лицензии по точкам.
- Брендпаки: спонсорские наборы движений.
- НЕ делать: энергию, обязательную рекламу в матче, платную фору.

## 6. UI / UX

### Решения (утверждено юзером 2026-07-08):
- ✅ **Арт-направление:** соревновательный (эволюция текущего неона), НО **сдержанно** — без
  вычурности и «игрушечности». Ориентир: чистая спортивная трансляция, а не RGB-перегруз.
- ✅ **Главный пользователь:** **взрослый-ведущий** (аниматор / учитель / родитель). Дети — участники
  на экране, а не операторы интерфейса.

### Опорная модель (предложено, ждёт подтверждения):
> **Телефон = пульт ведущего. Телевизор = шоу для детей.**
> Ведущий держит телефон как «пульт рефери» и управляет потоком; дети смотрят арену на большом экране.

### Архитектура экранов (host-first):
1. Session Home — «Быстрый матч» / «Турнир» / «Игроки» / «Настройки».
2. Roster (Игроки) — сохранённые дети (имя, цвет, аватар), переиспользуются между матчами.
   → решает боль «имена вводятся заново каждый матч».
3. Match Setup — выбор 2 игроков (или авто из сетки), режим, фора, длина раунда.
4. Calibration (есть) — оба в кадре, лок, отсчёт. Это то, что видят дети на ТВ.
5. Live Match — арена (соревновательный скин), которую смотрит зал. У ведущего минимум контролов (стоп/пауза).
6. Between-matches loop — победитель, фото-финиш, «Поделиться клипом», и КЛЮЧЕВОЕ: «Следующий матч» /
   «В сетку» в 1–2 тапа. Скорость прогона back-to-back матчей — главная ценность для аниматора.
7. Bracket view — турнирная сетка, тап продвигает победителя.

### Как решения меняют приоритеты:
- Roster (сохранённые игроки) поднимается в P0 — это ядро host-first.
- Bracket поднимается ближе к P0/раннему P1 — это главная ценность ведущего.

  1. *Матчер идентичности*: во время матча роли примагничены к телам (перестановочный матчинг

     и меняться местами без подмены очков. Ре-бинд потерянного игрока по близости в окне 2.5с,
     дальше позиционный фолбэк. Роли фиксируются с отсчёта 3-2-1 (`rolesLocked`).
  2. *ROI-зум*: когда оба захвачены, инференс идёт по кропу вокруг игроков (паддинг 35%) —

     Из-за смены системы координат встроенный трекер/сглаживание MoveNet выключены —
     идентичность и сглаживание полностью наши (EMA уже были).
  3. *Честный скоринг*: среднее смещение на точку вместо суммы — пропавшее запястье больше
     не режет очки ребёнку в мешковатой одежде; добавлены локти (6 точек вместо 4).
  4. *WebGPU* с фолбэком на WebGL: 1.5–3× быстрее инференс на поддерживаемых устройствах
